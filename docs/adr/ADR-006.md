# ADR-006: Groq API over Self-Hosted Ollama for LLM Inference

**Status:** Accepted
**Date:** 2026-06-27
**Deciders:** Platform Engineering

## Context

The prototype requires LLM inference for RAG answer generation. Two options were evaluated:
local inference via Ollama, and cloud inference via the Groq API.

## Decision

Use the Groq API (Llama 3.1 8B Instant) for LLM inference in the prototype.

## Rationale

| Factor | Ollama | Groq API |
|---|---|---|
| Hardware required | 8GB+ VRAM GPU or slow CPU | None |
| Throughput (8B model) | ~41 tokens/sec on M2 | ~800 tokens/sec |
| Setup complexity | Install + pull model (~4GB) | One API key |
| Cost (prototype) | Electricity + hardware | Free tier (14,400 req/day) |
| Latency (P99) | 673ms at low concurrency | ~200ms |
| Data privacy | Fully local | Data sent to Groq servers |

## Trade-offs

**Accepted:** Queries are sent to Groq's servers. For a prototype or internal deployment with
non-sensitive test data, this is acceptable. For a production deployment with real employee data,
data residency requirements may mandate switching to a self-hosted model.

**Migration path:** The `_call_llm` function in `pipeline.py` wraps all LLM calls. Switching
to vLLM or Ollama requires changing only that function and the model configuration in `.env`.

## Consequences

- No GPU required for the prototype
- First-time setup is one environment variable (`GROQ_API_KEY`)
- Production deployment with real PII must evaluate DPDP Act 2023 data-localisation requirements
- See `docs/GPU_REQUIREMENTS.md` for self-hosted hardware specifications
